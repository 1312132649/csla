RockfordLhotka replied on Wednesday, February 10, 2010

The .NET data portal doesn't offer the compression options provided by the SL data portal. This is because there are readily available WCF bindings that do compression, and it is preferable to use those bindings than the approach used in SL.

We only did what we did in the SL data portal because there were no compressed bindings for SL WCF when we created CSLA .NET for Silverlight. Sergey is in the process of providing an alternate proxy/host model for 3.8.3 and 4.0 that defers all the SL data portal processing to WCF too, because there are now good options for compressed bindings in SL3 - and the current SL data portal model prevents them from really working the way you'd expect...
