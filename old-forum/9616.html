Hi Rocky

I started to look into it in some more detail (just to try and gauge the effort required). It doesn't seem too laborious and mostly an initial effort to define a "Package Specification" that describes the library. This package specification and package contents is then submitted online, to enable developers to discover and consume the packages.

Put very simply, the creation of a package boils down to:

  • Create a NuSpec file (XML based file)
    • Define metadata for package
    • Define dependencies on other packages
    • Define contents of the package
      • Tools
      • Content - Samples / Source could potentially be specified here
      • Llb / Assemblies - The framework and version can be specified. For example: Bin\{framework name}{version} which could be:
        • \Bin\.NET2.0  --> .NET Framework version 2.0
        • \Bin\.NET4.0 --> .NET Framework version 4.0
        • \Bin\SL4 --> Silverlight 4.0
        • etc.
  • Build a package
    • This can be specified as Visual Studio Post-Build Event. Future versions of NuPack intend to create a MSBuild Task)
    • Use NuPack.exe to build a package. For example
      • "$(SolutionDir)Tools\NuPack.exe" "CSLA.NuSpec"
  • Submit
    • This is where I'm coming up empty. Not sure if I'm just not seeing it, but it appears that this is not yet available. I have noticed a couple of other users enquiring about it too, so I'm sure they'll get that information out soon.
    • It's worth noting that the source of packages is "feed" based and that while there is a "central" feed maintained by the community, you can also specify a custom feed which could even be local/internal feed (useful for corporate environments).

Here are some direct links to useful bits of info:

  • The CodePlex site for NuPack
  • Getting started - Using packages
  • Creating a package
  • NuSpec format
  • NuSpec FAQ
  • Current list of available packages

PS: I like that is has the ability to request a list of available updates for the packages already referenced (E.g. List-Package –updates).

Summary: At first glance it seems great and probably overdue for the VS environment, but it is still a bit v1.0 at this stage and there are a couple of small issues. I suspect it's going to gain momentum and with it some good polish to go with what is an already great start. I'm not particularly fond of where it places the downloaded package and files in your solution / project nor the fact that you cannot control it (yet). It might also just be a case of resistance to change from my own practices.

I'll try to keep following it and update here if I get time.

Jaans
